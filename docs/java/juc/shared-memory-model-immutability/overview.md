# 共享模型之不可变

前面章节学习了使用**锁机制**和**无锁编程**保证线程安全，但它们都需要处理共享可变状态的并发访问问题。

本章将学习另一种线程安全策略——**不可变对象设计**，通过消除可变状态从根本上避免并发问题。

## 学习目标

通过本章学习，你将掌握：

- 理解**不可变对象**的设计原则与实现方式
- 掌握 **final 关键字**的底层原理与内存语义
- 学习**享元模式**在 JDK 中的应用
- 掌握**无状态设计**实现天然线程安全
- 理解不可变对象在并发编程中的价值

## 不可变设计的优势

### 线程安全
- **无需同步**：不可变对象天然线程安全，无需加锁
- **简化并发**：不用担心状态被其他线程修改
- **安全共享**：可以安全地在多个线程间共享

### 性能优势
- **无锁开销**：避免 synchronized 和 Lock 的性能损耗
- **缓存友好**：可以安全地缓存不可变对象（如 String 常量池）
- **无竞争**：多线程访问无需竞争

### 设计简洁
- **易于理解**：对象状态不变，代码逻辑更清晰
- **易于测试**：无状态变化，测试用例更简单
- **防御性**：可作为 HashMap 的 key，不会因修改导致问题

## 不可变设计的权衡

### 创建开销
- 每次修改都需要创建新对象
- 频繁修改会产生大量临时对象
- 增加垃圾回收压力

### 适用场景

| 场景 | 是否适合不可变设计 |
|------|-------------------|
| 配置对象、DTO | ✅ 适合 |
| 多线程共享的数据 | ✅ 适合 |
| 作为 Map 的 key | ✅ 适合 |
| 频繁修改的业务对象 | ❌ 不适合 |
| 大对象且需频繁创建 | ❌ 不适合 |
